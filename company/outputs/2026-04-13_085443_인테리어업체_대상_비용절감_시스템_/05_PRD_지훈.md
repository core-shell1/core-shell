# PRD: 인테리어 업체 대상 표준예가 견적서 자동 산출 SaaS

## 제품 개요
중소 인테리어 업체(직원 5~30명) 사장이 현장개요(평수·용도·기존 상태)만 입력하면 1시간 내 표준예가 견적서·자재 상세내역서를 자동 산출하는 B2B SaaS 솔루션 (월 29만~49만 원 구독).

---

## 기술 스택

- **FE**: React 18 + TypeScript / Tailwind CSS (반응형 웹앱, PC 중심 + 모바일 보조)
- **BE**: Node.js (Express) + TypeScript / PostgreSQL (관계형 DB, 견적 데이터 정규화 필요)
- **DB**: PostgreSQL (견적·자재·고객 데이터 ACID 보장) + Redis (세션·캐시, 견적 산출 성능 최적화)
- **인프라**: AWS EC2 (애플리케이션 서버) + RDS (PostgreSQL 관리형) + S3 (견적서 PDF/엑셀 저장) + CloudFront (CDN, 파일 배포 가속화)
- **외부 API**: 조달청 표준품셈 API (자재 단가 데이터) + 건설공사 표준시장단가 API (지역별 인건비 데이터)
- **배포**: GitHub Actions (CI/CD) → AWS EC2 자동 배포

## Must NOT (범위 외)

- **건축 허가/인허가 자동 신청** — 법적 책임 문제, 지자체별 규정 상이 → 제외
- **시공 현장 관리 시스템 (공정 추적/하자 관리)** — 범위 확대, 경쟁사(아정당) 기능과 중복 → P2 이후 검토
- **클라이언트 직접 결제 기능** — B2B 모델(사장 결제)이므로 B2C 결제 불필요 → 제외
- **모바일 앱 (iOS/Android 네이티브)** — 초기 웹앱으로 충분, 모바일 반응형으로 대체 → P2 이후 검토
- **다국어 지원** — 초기 한국어만, 지역 자재 단가 데이터 한국 기준 → P2 이후 검토
- **AI 기반 공사비 예측 (머신러닝)** — 초기 규칙 기반 알고리즘으로 충분, 데이터 축적 후 P2 진행
- **ERP 연동 (SAP/더존 등)** — 초기 엑셀 다운로드로 충분, P2 이후 API 제공

## User Flow

### 시나리오 1: 신규 고객 첫 견적 산출 (Happy Path)

1단계: **회원가입** → 사용자가 랜딩페이지에서 "무료 체험 시작" 클릭 → 이메일/휴대폰 입력 → 인증 완료 (1분)

2단계: **현장개요 입력** → 대시보드 접근 → "새 견적 만들기" 클릭 → 폼 입력 (평수 30평, 용도 아파트, 기존 상태 리모델링, 예산 3,000만 원, 지역 서울) (5분)

3단계: **견적서 미리보기** → 입력 완료 후 "견적서 생성" 클릭 → 표준예가 견적서 PDF 1페이지 미리보기 (공정별 인건비·자재비 분리 표시) (30초)

4단계: **정확도 체감 (아하 모먼트)** → 사용자가 "오차율 ±10% 정도면 충분하네" 판단 → 신뢰도 상승

5단계: **유료 전환** → "전체 견적서 다운로드" 클릭 → "집중 플랜 가입 필요 (월 49만, 첫 달 50% 할인 24.5만)" 팝업 → 결제 완료 (토스페이먼츠) (2분)

6단계: **견적서 다운로드** → 엑셀 파일 다운로드 → 자재 상세내역서 자동 생성 (품목·수량·단가) → 클라이언트에게 이메일 전달

7단계: **재사용** → 다음 주 새로운 현장 견적 요청 → 2~3단계 반복 (월 3건 시공 가정 시 월 3회 사용)

---

### 시나리오 2: 기존 고객 월간 반복 사용

1단계: **로그인** → 사용자가 대시보드 접근 → "새 견적 만들기" 클릭

2단계: **현장개요 입력** → 이전 입력 기록 자동 제안 (지역·용도 등) → 새 현장 정보 입력 (3분)

3단계: **견적서 자동 생성** → "견적서 생성" 클릭 → 표준예가 견적서 + 자재 상세내역서 자동 산출 (1분)

4단계: **다운로드 + 클라이언트 전달** → 엑셀 다운로드 → 클라이언트 이메일 전달

5단계: **월간 사용 현황 확인** → 대시보드에서 "월 견적 산출 건수 3/30" 표시 (집중 플랜 월 30건 제한)

---

### 시나리오 3: 정확도 불만족 시 환불 (Edge Case)

1단계: **견적서 다운로드 후 현장 방문** → 사용자가 실제 공사 비용 확인 → 견적 오차율 20% 발생 (예: 견적 3,000만 → 실제 3,600만)

2단계: **환불 요청** → 대시보드 "고객 지원" → "정확도 보증 환불 신청" 클릭 → 사유 입력 (오차율 15% 초과)

3단계: **환불 처리** → CS팀 검토 (1일) → 환불 승인 → 토스페이먼츠 자동 환불 (3일 내)

4단계: **서비스 계속 사용** → 환불 후에도 30일 무료 체험 기간 유지 (신뢰 확보)

## 성공 기준

### OKR (6개월 기준)

- **KR1**: 베타 6개월 내 유료 전환 30곳 확보 (MRR 300만 원 이상)
  - **측정 방법**: 구독 관리 대시보드에서 `subscriptions.status = 'active'` 카운트 (월별 추적)
  - **목표 달성 기준**: 6개월 말 활성 구독 30곳 이상 (월 평균 5곳 신규 전환)
  - **실패 임계값**: 3개월 말 10곳 미만 → 채널/카피 개선 필요

- **KR2**: 견적서 산출 정확도 오차율 15% 이내 달성 (10건 이상 검증)
  - **측정 방법**: 베타 사용자가 실제 공사 비용 입력 후 오차율 계산 (`(실제비용 - 견적비용) / 실제비용 × 100`)
  - **목표 달성 기준**: 10건 이상 검증 시 평균 오차율 ±15% 이내 (±10% 목표)
  - **실패 임계값**: 오차율 20% 초과 → 표준예가 DB 정확도 개선 필요

- **KR3**: 거래망 추천 기능 통해 중개 수수료 월 50만 원 이상 발생 (부가 수익 검증)
  - **측정 방법**: 거래망 추천으로 성사된 거래 건수 × 5% 수수료 (월별 추적)
  - **목표 달성 기준**: 월 거래 성사 200건 이상 (거래 평균 500만 원 × 5% = 25만 × 200건 = 5,000만 원)
  - **실패 임계값**: 거래 성사 50건 미만 → 거래망 DB 부족 또는 신뢰도 낮음 → P2 기능으로 연기

## 리스크

### 1. 기술 리스크

**리스크 1-1: 표준예가 DB 정확도 부족 (오차율 15% 초과)**
- **근거**: 조달청 표준품셈 + 건설공사 표준시장단가 공공데이터가 지역별·시점별 편차 큼 (서울 vs 지방, 2023년 vs 2024년 가격 상이)
- **영향**: 견적 오차율 15% 초과 시 고객 신뢰 붕괴 → 환불율 10% 초과 → 단위경제 악화
- **대응 방안**:
  1. **MVP 단계**: 조달청 데이터 + 기존 인테리어 업체 3곳의 과거 견적 데이터 수집 (100건 이상) → 하이브리드 모델로 정확도 향상
  2. **베타 단계**: 사용자 피드백 기반 알고리즘 재학습 (실제 공사비 입력 → 모델 개선) → 3개월 내 오차율 ±10% 달성 목표
  3. **정책**: "정확도 보증 30일 환불" 정책으로 초기 신뢰 확보 → 데이터 축적 후 정책 강화

**리스크 1-2: AI 도면 분석 개발 난이도 높음 (P1 기능, 개발 난이도 8/10)**
- **근거**: 도면 이미지 → 2D 평면도 자동 생성은 컴퓨터 비전 + 머신러닝 필요 (초기 정확도 70% 목표)
- **영향**: 개발 기간 6개월+ 소요 → 현금 소진 가속화 → MVP 출시 지연 가능
- **대응 방안**:
  1. **MVP 범위 축소**: 도면 기능 제외, 현장개요 텍스트 입력만으로 견적 산출 (개발 난이도 3/10)
  2. **P1 단계**: 도면 기능은 3개월 후 추가 (초기 고객 확보 후 수익으로 개발비 충당)
  3. **외주 검토**: 도면 분석 API (예: Roboflow, Google Vision API) 활용 → 자체 개발 대비 6주 단축

**리스크 1-3: 조달청 API 접근 제한 또는 데이터 품질 저하**
- **근거**: 조달청 표준품셈 API가 공공데이터이나 업데이트 주기 불명확 (연 1회? 분기 1회?)
