# PRD: 거래망 추천 시스템 (Trade Partner Recommendation Engine)

**문서 버전:** v1.0  
**작성자:** 지훈 (실행팀 PRD 담당)  
**최종 승인:** 준혁 (조건부 GO)  
**작성일:** 2025년 1월  
**상태:** 개발 시작 가능

---

## 제품 개요

**한 줄:** 신용도 기반 거래망 추천으로 인테리어 시공팀장/구매 담당자의 거래 리스크를 줄이고 수주율을 높이는 B2B SaaS.

**핵심 가치 명제:**
- **Before:** 거래처 신용도 확인 = 수동 조사 + 기존 네트워크 의존 → 리스크 높음, 시간 낭비
- **After:** 신용도 기반 거래망 추천 시스템 → 1초 내 안전한 거래처 발견 → 연체 손실 감소, 수주율 증가

**타겟 페르소나:**
- 1차: 인테리어 시공팀장 / 구매 담당자 (연 거래액 5억~10억 원 규모)
- 2차: 소규모 인테리어 시공업체 대표 (자재·하청 거래처 관리)

**가격 (초기):** 월 20만 원 (Starter 플랜)

## 기술 스택

| 계층 | 선택 | 근거 |
|------|------|------|
| **FE** | React 18 + TypeScript + Tailwind CSS | 빠른 프로토타입, 리안 기존 스택 활용 |
| **BE** | Node.js (Express) + TypeScript | 빠른 개발, 비동기 처리 최적화 |
| **DB** | PostgreSQL (주) + Redis (캐시) | 관계형 데이터 (거래처, 신용도), 빠른 조회 |
| **인프라** | AWS EC2 + RDS + S3 | 스케일 가능, 비용 효율적 |
| **인증** | JWT + OAuth 2.0 (선택사항) | B2B 기업 계정 관리 |
| **외부 API** | 나이스평가정보 / KCB (신용도 조회) — Phase 2 | MVP는 수동 DB로 대체 |
| **배포** | GitHub Actions + Docker | CI/CD 자동화 |

**개발 환경:**
- Node.js 18+
- npm / yarn
- PostgreSQL 14+
- Redis 7+

## 기능 목록

> **Wave 구조:**
> - **Wave 1 (P0):** MVP 출시 (3개월) — 거래망 추천 핵심만
> - **Wave 2 (P1):** 1차 확장 (3~6개월) — 신용도 고도화 + 법률 리스크 체크
> - **Wave 3 (P2):** 2차 확장 (6~12개월) — 공기 산출, 사업성 평가서

### Wave 1 (P0) — MVP: 거래망 추천 핵심

| 우선순위 | 기능 | 설명 | 이유 | 의존 기능 | 예상 개발 기간 |
|---------|------|------|------|---------|-------------|
| **P0** | 회원가입 / 로그인 | 이메일 + 비밀번호 기반 인증 | 기본 기능 | 없음 | 3일 |
| **P0** | 기업 정보 입력 (프로필) | 회사명, 업종(시공/자재/하청), 거래액 범위 입력 | 추천 알고리즘 기초 데이터 | 회원가입 | 2일 |
| **P0** | 거래망 추천 (코어) | 입력된 기업 정보 기반 신용도 순 거래처 추천 (상위 10개) | **Pain Level 5** — 거래 리스크 감소 | 기업 정보 입력 | 7일 |
| **P0** | 추천 거래처 상세 정보 | 추천 거래처 신용도 점수, 거래 이력, 연락처 표시 | 의사결정 근거 제공 | 거래망 추천 | 3일 |
| **P0** | 추천 거래처 저장 / 즐겨찾기 | 마음에 드는 거래처 저장 → 대시보드에서 관리 | 반복 접근 편의성 | 거래망 추천 | 2일 |
| **P0** | 대시보드 (기본) | 저장된 거래처 목록, 최근 추천 이력 표시 | 사용자 중심 정보 제공 | 거래망 추천 | 3일 |
| **P0** | 신용도 DB (수동 큐레이션) | 초기 50개 거래처 신용도 점수 수동 입력 (엑셀 기반) | **필수 조건** — API 연동 전까지 서비스 성립 | 없음 | 운영팀 (개발 외) |
| **P0** | 결제 / 구독 관리 | 월 20만 원 구독, 카드 결제 (PG: 토스페이먼츠) | 수익화 | 회원가입 | 5일 |
| **P0** | 메일 알림 (기본) | 가입 확인, 추천 거래처 업데이트 알림 | 재방문 유도 | 회원가입 | 2일 |

**Wave 1 총 개발 기간:** ~3주 (병렬 개발 기준)  
**Wave 1 완성 기준:** 첫 유료 고객 5명 이상 결제 확인

| 우선순위 | 기능 | 설명 | 이유 | 의존 기능 | 예상 개발 기간 |
|---------|------|------|------|---------|-------------|

| **P1** | 법률 리스크 체크 | 거래처 소송/부도 이력 자동 검색 (공개 정보 기반) | **Pain Level 5** — 법적 리스크 사전 차단 | 거래망 추천 | 7일 |
| **P1** | 거래처 비교 기능 | 2~3개 거래처 신용도/리스크 나란히 비교 | 의사결정 속도 향상 | 거래망 추천 | 3일 |

| **P1** | 팀 초대 / 권한 관리 | 같은 회사 팀원 초대, 역할별 권한 설정 | B2B 협업 | 회원가입 | 5일 |

**Wave 2 총 개발 기간:** ~4주  
**Wave 2 완성 기준:** 월 매출 1,000만 원 달성, 유료 고객 50명 이상

## Must NOT (범위 외)

| 제외 기능 | 이유 |
|---------|------|
| **렌더링 / 3D 설계 도구** | 핵심 pain point 아님. 기술 난이도 높음. Wave 1 범위 초과. |
| **자재 가격 비교** | 거래망 추천과 무관. 별도 서비스로 분리 필요. |
| **공사 일정 관리 (Gantt)** | 프로젝트 관리 도구와 중복. 거래망 추천 가치와 무관. |
| **AI 기반 설계 제안** | 기술 난이도 극도로 높음. 초기 고객 확보와 무관. |
| **모바일 앱 (Native)** | Wave 1은 웹 전용. 반응형 디자인으로 모바일 대응. Native 앱은 Wave 3 이후. |
| **다국어 지원** | 초기 타겟은 한국 시장만. 글로벌 확장 후 고려. |
| **오프라인 결제 (계좌이체 자동화)** | 초기는 카드 결제만. 계좌이체는 Wave 2 이후. |

## Aha Moment 정의

**Aha Moment (가치 깨닫는 순간):**
> "거래처 신용도 점수를 보고 '아, 이 업체는 안 되겠다'고 판단하는 순간"

**측정 지표:**
- 가입 → 첫 추천 거래처 조회까지: **30초 이내** (목표)
- 추천 거래처 상세 정보 클릭 → 저장까지: **15초 이내**
- **목표:** 가입부터 Aha까지 **60초 이내**

**검증 방법:**
- 초기 고객 10명 사용 시간 측정 (Hotjar / LogRocket)
- "이 서비스 쓸 만하다"고 느낀 시점 인터뷰

## Customer Forces Strategy (서윤 Phase 3 Canvas 기반)

### Push 요인 (현재 상황의 고통)
- **경쟁사 불만:** 기존 거래처 관리는 엑셀 + 수동 조사 → 시간 낭비, 신용도 정보 부정확
- **현재 상황의 긴급성:** 거래 리스크로 인한 연체 손실 (평균 500만 원/년)

### Pull 요인 (우리 서비스의 차별 가치)
- **신용도 기반 추천:** 나이스평가정보 API 기반 실시간 신용도 조회 (경쟁사 없음)
- **법률 리스크 자동 체크:** 소송/부도 이력 자동 검색 (원스톱)
- **빠른 의사결정:** 30초 내 거래처 판단 가능

### Inertia 낮추기 (마이그레이션 장벽 제거)
- **기능으로 변환:**
  - 기존 거래처 일괄 업로드 기능 (CSV import)
  - 엑셀 내보내기 (기존 워크플로우 유지)
  - 온보딩 영상 (3분) — 첫 추천까지 가이드
  - 무료 트라이얼 (7일, 제한 없음)

### Anxiety 해소하기 (신뢰 신호)
- **기능으로 변환:**
  - 신용도 점수 근거 명시 (어느 데이터 기반인지 투명 공개)
  - 거래처 평점 시스템 (다른 사용자 평가 공개)
  - 환불 보장 (30일 이내 만족 못 하면 전액 환불)
  - 초기 고객 사례 (시공팀장 인터뷰 영상)

## User Flow

### 시나리오 1: 신규 사용자 — 첫 거래처 추천받기

```
1단계: 랜딩 페이지 방문 → "지금 시작하기" 클릭
   ↓ 시스템 응답: 회원가입 폼 표시

2단계: 이메일 + 비밀번호 입력 → 가입 완료
   ↓ 시스템 응답: 확인 메일 발송, 대시보드로 리다이렉트

3단계: 기업 정보 입력 (회사명, 업종, 연 거래액)
   ↓ 시스템 응답: 입력 정보 저장, "추천 받기" 버튼 활성화

4단계: "추천 받기" 클릭
   ↓ 시스템 응답: 신용도 순 거래처 상위 10개 표시 (2초 이내)

5단계: 거래처 상세 정보 클릭 (신용도 점수, 거래 이력, 연락처)
   ↓ 시스템 응답: 상세 정보 팝업 표시

6단계: "저장" 버튼 클릭
   ↓ 시스템 응답: 즐겨찾기에 추가, 대시보드에 표시

7단계: 결제 페이지로 이동 (월 20만 원 구독)
   ↓ 시스템 응답: 카드 결제 완료, 구독 활성화
```

**Aha Moment:** 4단계 — 신용도 점수를 보고 "이 업체는 괜찮겠네" 판단

---

### 시나리오 2: 기존 사용자 — 거래처 비교하기

```
1단계: 대시보드 접속
   ↓ 시스템 응답: 저장된 거래처 목록 + 최근 추천 이력 표시

2단계: "새 추천 받기" 클릭 + 필터 설정 (지역, 신용도 범위)
   ↓ 시스템 응답: 필터링된 거래처 10개 표시

3단계: 2~3개 거래처 선택 → "비교하기" 클릭
   ↓ 시스템 응답: 신용도, 법률 리스크, 거래 이력 나란히 표시

4단계: 최종 선택 → "연락처 복사" or "메일 발송"
   ↓ 시스템 응답: 연락처 클립보드 복사 or 이메일 템플릿 생성
```

---

### 시나리오 3: 팀 협업

```
1단계: 대시보드 → "팀 초대" 클릭
   ↓ 시스템 응답: 초대 링크 생성

2단계: 팀원 이메일 입력 → 역할 선택 (관리자/멤버)
   ↓ 시스템 응답: 초대 메일 발송

3단계: 팀원이 초대 수락
   ↓ 시스템 응답: 같은 거래처 목록 공유, 협업 가능
```

| 화면명 | URL/Route | 핵심 컴포넌트 | 동작 |
|--------|-----------|-------------|------|
| **랜딩 페이지** | `/` | 헤더 + 히어로 섹션 + 기능 소개 + CTA ("지금 시작하기") | 클릭 → `/auth/signup`으로 이동 |

## 리스크

### 1. 가치 실패 (Value Risk) — **하은 반론 기반**

**리스크:** 거래망 추천이 실제로 수주율을 높이지 못함. 사용자가 "신용도 정보는 좋은데, 결국 기존 거래처 쓴다"고 판단.

**근거 (하은):** 서윤의 인터뷰 중 "거래 리스크"는 Level 5이지만, 실제 Workaround 비용(연체 손실 500만 원/년)이 서비스 가격(월 20만 원 = 연 240만 원)과 비교해 충분히 크지 않을 수 있음. 또한 인테리어 업체는 이미 기존 거래처 네트워크에 의존하는 문화가 강해, 새로운 거래처 발굴 자체를 원하지 않을 수 있음.

**대응 방안:**
- **48시간 검증 필수:** 인테리어 업체 10명에게 직접 전화 — "지금 자재 거래처 어떻게 찾으세요? 신용도 모르는 업체랑 거래하다 손해 본 적 있으세요?" (Problem Interview)
- **성공 기준:** 7명 이상 "거래망 리스크로 실제 손실 경험 있음" 확인 필수. 미달 시 즉시 2순위(법률 리스크 체크)로 피벗.
- **초기 고객 선정:** 첫 10명은 "거래처 신용도 때문에 손실 본 경험이 있는" 사람만 타겟. 일반 시공팀장은 나중에.

---

### 2. 전환 실패 (Conversion Risk) — **하은 반론 기반**

**리스크:** B2B 인테리어 업체의 SaaS 구독 전환율이 극도로 낮음. 이 업종은 "월 구독" 문화가 약하고, "건당 과금" or "연간 계약"을 선호할 수 있음.

**근거 (하은):** 민수의 CAC 5만 원은 B2C 기준. 인테리어 B2B 아웃리치 현실 CAC는 20~50만 원 수준. LTV 50만 원(월 20만 원 × 2.5개월 평균 유지)이면 LTV/CAC 1~2.5로 손익분기 위험.

**대응 방안:**
- **가격 모델 재검토:** 초기 고객 5명 확보 후, 실제 결제 의사 조사 — "월 20만 원 구독 vs 건당 5만 원 과금 vs 연 200만 원 선결제" 중 선호도 조사.
- **선결제 유도:** 초기 고객에게 "3개월 선결제 시 50% 할인(월 10만 원)" 오퍼 → 현금 흐름 개선 + 전환율 향상.
- **CAC 재산정:** 실제 마케팅 채널별 CAC 측정 후, LTV/CAC ≥ 3 달성 불가능하면 가격 인상 or 채널 변경.

---

### 3. 도달 실패 (Reach Risk) — **관리 가능**

**리스크:** 인테리어 업체에 도달할 수 있는 마케팅 채널이 제한적. 네이버 카페, 건설업 협회 등 오프라인 채널 의존도 높음.

**근거:** 리안 컴퍼니의 기존 고객(뷰티/카페 소상공인)과 인테리어 업체는 다른 커뮤니티에 속함. 기존 마케팅 채널 재활용 불가.

**대응 방안:**
- **초기 채널 3가지 동시 실행:**
  1. 네이버 카페 (인테리어 시공 커뮤니티) — 유기 아웃리치
  2. 건설업 협회 / 시공업체 협회 — 파트너십 or 직접 DM
  3. 기존 고객 추천 (인테리어 업체 지인) — 리퍼럴
- **초기 100명 도달 목표:** 3개월 내 위 3가지 채널로 100명 가입 달성 가능 (현실적 추정).
- **채널별 CAC 측정:** 각 채널의 실제 CAC 계산 후, 가장 효율적인 채널에 집중.

---

### 4. 단위경제 실패 (Unit Economics Risk) — **하은 반론 기반**

**리스크:** 민수의 LTV/CAC 5만 원은 근거 빈약. 실제 인테리어 B2B 환경에서는 CAC가 훨씬 높을 수 있음.

**근거 (하은):**
- CAC 5만 원 = 채널별 CPC × 전환율 역산 (근거 없음)
- 인테리어 B2B는 구글 광고 CPC가 높고, 전환율도 낮음
- 현실적 CAC: 20~50만 원 (아웃리치 + 영업 비용 포함)
- LTV 50만 원 (월 20만 원 × 2.5개월) vs CAC 50만 원 = LTV/CAC 1 (손익분기)

**대응 방안:**
- **CAC 재산정 필수:** 초기 고객 5명 확보 후, 실제 마케팅 비용 / 신규 고객 수 계산.
- **LTV 향상 전략:**
  - 월 구독 → 연간 계약 유도 (선결제 할인)
  - 업셀 (Pro 플랜 20만 원 → 35만 원)
  - 추가 기능 (법률 리스크 체크, 공기 산출) 번들링
- **CAC 절감 전략:**
  - 유기 채널 (네이버 카페, 커뮤니티) 우선
  - 리퍼럴 프로그램 (기존 고객 추천 시 할인)
  - 파트너십 (건설업 협회, 자재 유통사 협력)
- **손익분기 조건:** LTV/CAC ≥ 3 달성 불가능하면, 가격 인상 or 기능 축소 (비용 절감).

---

### 5. 신용도 DB 구축 실패 (Data Risk) — **필수 조건**

**리스크:** 신용도 DB 없이는 서비스 자체가 성립 안 됨. 나이스평가정보 API 연동 전까지 수동 큐레이션으로 초기 50개 거래처 DB 구축해야 하는데, 이 과정에서 시간/비용 소모 가능.

**근거:** 거래망 추천 = 신용도 기반 매칭. 신용도 데이터 없으면 "추천"이 아닌 "랜덤 리스트" 제공.

**대응 방안:**
- **초기 DB 구축 전담:** 운영팀 1명이 인테리어 업체 50개 직접 인터뷰 + 거래 이력 수집 (2주 소요).
- **데이터 소스:**
  - 기존 고객 네트워크 (리안 컴퍼니 인테리어 고객)
  - 네이버 카페 (인테리어 시공 커뮤니티) 활동 업체
  - 건설업 협회 회원사
  - 자재 유통사 거래처 리스트
- **신용도 점수 산정 기준:**
  - 거래 이력 (연체 여부, 거래액)
  - 사업자등록 정보 (폐업 여부)
  - 온라인 평판 (네이버 블로그, 카페 언급)
  - 수동 점수 (0~100)
- **Wave 2 이후:** 나이스평가정보 API 연동 → 자동 신용도 조회로 전환.

### 6. 경쟁사 진입 (Competition Risk) — **관리 가능**

**리스크:** 현재 블루오션이지만, 반장창고·스팬딧 등 기존 건설 SaaS가 기능 확장 시 위협 가능.

**근거:** 거래망 추천 + 법률 리스크 + 공기 산출 원스톱은 국내 경쟁사 없음. 하지만 선점 속도가 느리면 경쟁사가 따라올 수 있음.

**대응 방안:**
- **선점 속도 최우선:** MVP 3개월 내 출시 → 초기 고객 100명 확보 → 데이터 축적 (신용도 DB, 거래 이력).
- **Moat 구축:**
  - 신용도 DB 확대 (1,000개 → 10,000개 거래처)
  - 거래 이력 데이터 축적 (네트워크 효과)
  - 거래처 평점 시스템 (커뮤니티 신뢰도)
- **차별화 전략:**
  - 법률 리스크 자동 체크 (경쟁사 대비 선행)
  - 공기 산출 (표준예가 기반 자동화)
  - 사업성 평가서 (의사결정 자동화)

---

### 7. 기술 난이도 (Technical Risk) — **관리 가능**

**리스크:** 전체 기능 스택(렌더링+법률+DB+견적) 동시 구현은 스타트업 수준 불가. 범위 초과로 인한 개발 지연 가능.

**근거:** 준혁의 기술 난이도 평가 = 7/10 (높음). 거래망 추천 단독 = 3/10 (낮음).

**대응 방안:**
- **범위 강제 축소:** 지금 당장 렌더링·법률리스크·공기산출·사업성평가서는 로드맵에서 제거. **거래망 추천 하나만** MVP로 정의.
- **개발 리소스:** 개발자 2명 (FE 1명, BE 1명) 3개월 집중 → MVP 완성.
- **외주 활용:** 필요 시 UI/UX 디자인 외주 → 개발 속도 향상.
- **Wave 단계별 확장:** Wave 1 (거래망) → Wave 2 (신용도 고도화) → Wave 3 (공기 산출) 순서로 단계적 확장.

---

### 8. 시장 긴급성 부족 (Market Urgency Risk) — **관리 가능**

**리스크:** "지금 당장 안 쓰면 망한다"는 긴박감이 약함. 인테리어 업체는 기존 거래처로 충분하다고 생각할 수 있음.

**근거:** 거래 리스크는 현재진행형이지만, 대부분의 업체는 "이 정도는 감수할 수 있다"고 생각.

**대응 방안:**
- **Pain Point 강조:** 초기 마케팅에서 "연체 손실 500만 원/년" 구체적 수치 제시.
- **사례 제시:** 초기 고객 사례 (시공팀장 인터뷰 영상) → 신뢰도 향상.
- **무료 트라이얼:** 7일 무료 사용 → 가치 체험 후 결제 유도.
- **환불 보장:** 30일 이내 만족 못 하면 전액 환불 → 구매 장벽 낮춤.

---

### 9. SOM 과대 추정 (Market Sizing Risk) — **하은 반론 기반**

**리스크:** 민수의 SOM 2,000~3,000만 달러는 글로벌 "인테리어 디자인 소프트웨어" 시장 전체. 한국 소규모 인테리어 업체 대상 실제 SOM은 훨씬 작을 수 있음.

**근거 (하은):** 
- 글로벌 시장 ≠ 한국 소규모 업체 시장
- 한국 인테리어 시공업체 수: 약 50,000개 (통계청)
- 타겟 (연 거래액 5억~10억 원): 약 5,000개 (10%)
- 월 20만 원 × 5,000개 = 월 10억 원 = 연 120억 원 (SOM)

**대응 방안:**
- **SOM 재산정:** 한국 시장 기준 실제 SOM = 50~200억 원 (보수적 추정).
- **TAM 축소:** 초기 타겟 = 서울/경기 인테리어 시공업체 (약 1,000개) → 월 2억 원 = 연 24억 원.
- **SAM 설정:** 초기 3년 목표 = 시장 점유율 5% → 월 1,000만 원 = 연 12억 원.
- **과대 추정 금지:** 모든 재무 예측은 보수적 추정 기반.
