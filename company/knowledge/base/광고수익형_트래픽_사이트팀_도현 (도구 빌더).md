### UX design best practices for free online calculator and converter tools that maximize time on site and ad impressions
### 핵심 UX 베스트 프랙티스: 시간 지연 & 광고 노출 극대화

무료 온라인 계산기/컨버터 도구의 UX는 **단순성 + 개인화 + 즉시성**으로 사용자 이탈을 막고, 입력 조정 반복(재계산)을 유도해 **페이지 체류시간 2-3배 증가**, **광고 노출 30-50%↑** 달성.[6][5]

#### 1. **임베드 & 즉시 로드 (접근성 1위)**
   - 전용 페이지 대신 **페이지 내 임베드**로 배치: 팝업 피하고 스크롤 중 로드 지연 없애기 (로드 시간 <2초).[6]
   - **SEO 최적화**: 키워드 타이틀/메타에 "무료 BMI 계산기" 등 삽입, 검색 유입 40%↑.[6]
   - 사례: NN/g 연구에서 임베드 도구 사용률 70% vs. 별도 페이지 30%.[6]

#### 2. **입력 최소화 + 조건부 표시 (개인화로 반복 유도)**
   - **필수 입력만 요구** (3-5개): 필드당 출력 변화 없으면 삭제, 드롭다운/슬라이더 우선 (정밀도 불필요 시).[6][7][5]
   - **조건부 로직**: 이전 입력 따라 필드 숨김/표시 (e.g., BMI 입력 후 체중 단위 자동 변경), 개인화로 재입력 2배.[5]
   - 입력 도움: 툴팁/예시/설명 버튼 (e.g., "월 소득: 세전 기준"), 학습 곡선 50%↓.[6][3]
   - 사례: ConvertCalculator, 필드 7개 줄여 완료율 65%↑.[7]

#### 3. **실시간 결과 + 쉬운 재조정 (체류시간 극대화)**
   - **입력 즉시 결과 업데이트** (디바운스 200ms): 엔터/버튼 없이 실시간 피드백, 재계산 루프 유발.[6][1]

   - 출력 **맥락화**: 숫자+그래프/비교표 (e.g., "당신 BMI 25: 과체중, 표준 범위 18.5-24.9"), 공유 버튼 추가.[6][3]

#### 4. **모바일/반응형 + 인터랙티브 (전체 트래픽 60% 커버)**
   - **터치 우선**: 버튼 >48x48px, 슬라이더/드래그 지원, 키보드 불필요.[1]

### progressive web app PWA for tool sites performance optimization Core Web Vitals LCP FID CLS improvement techniques
### **PWA 툴 사이트 성능 최적화: Core Web Vitals 핵심 지표 개선 프레임워크**

PWA(Progressive Web App) 툴 사이트(도구 중심 웹앱)는 **Service Worker 캐싱 + 앱 셸 아키텍처**로 초기 로드 50% 단축, Core Web Vitals(LCP/INP/CLS) 75% 사이트가 기준 미달 → 다음 7가지 기법 적용으로 **LCP 2.5s 이하, INP 200ms 이하, CLS 0.1 이하** 달성.[1][5]

#### **1. LCP (Largest Contentful Paint) <2.5s: 메인 콘텐츠 40% 사이트 실패 원인 TOP3 해결**
- **LCP 이미지 우선 로드**: `<img fetchpriority="high" loading="eager">` 사용, `data-src` 피함 (7% 페이지 지연).[1]
- **Preload 핵심 리소스**: `<link rel="preload" as="image" fetchpriority="high" href="lcp-hero.jpg">` + Speculation Rules API로 다음 페이지 사전 렌더링 (서버 낭비 0).[1]
- **비중요 리소스 지연**: JS/CSS/iframe 문서 끝 이동 or `loading="lazy"`, CDN+WebP 이미지 (파일 30%↓).[2][8]
- **PWA 특화**: Service Worker로 앱 셸(공통 UI) 캐싱, SSR/HTTP/2 서버 응답 100ms 목표.[3][2]

**사례**: 기사 메인 이미지 LCP 4s → preload+CDN 적용 1.8s (Chrome UX 40% 개선).[5][1]

#### **2. INP (Interaction to Next Paint, FID 후속) <200ms: 긴 JS 작업 80% 원인 분할**
- **긴 작업 분할**: Scheduler API `scheduler.yield()`로 기본 스레드 양보, 최대 50ms 단위 쪼개기.[1]
- **불필요 JS 최소화**: 코드 분할 `React.lazy(() => import('./module'))` + Tree Shaking (번들 40%↓), 태그 매니저 최적화.[1][7]
- **PWA Service Worker 캐싱**: 네트워크 요청 캐시 (Cache Header + Prefetch 폰트/CSS).[6][8]

**사례**: JS 과부하 INP 400ms → yield+분할 150ms, Lighthouse 90점.[1][7]

#### **3. CLS (Cumulative Layout Shift) <0.1: 시각 불안정 0으로**
- **예약 공간 확보**: `<img width="300" height="200">` + `aspect-ratio: 16/9` CSS, 폰트 `font-display: swap`.[1]
- **동적 삽입 방지**: 애니메이션 레이아웃 유도 CSS 피함, bfcache 활성화 (뒤로가기 즉시).[1]
- **PWA 앱 셸**: 정적 셸 고정, 동적 콘텐츠 lazy load.[3]

**측정 도구**: Chrome DevTools Performance 탭 + Web Vitals 확장 (실시간 점수).[10][5]
