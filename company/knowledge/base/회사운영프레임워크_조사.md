### 1-2. 각 프레임워크 핵심 요약

#### EOS (Entrepreneurial Operating System / Traction)
- **창시자**: Gino Wickman, 20만+ 기업이 도입
- **6대 요소**: Vision(비전), People(사람), Data(데이터), Issues(이슈), Process(프로세스), Traction(실행력)
- **핵심 도구**:
  - **V/TO (Vision/Traction Organizer)**: 회사 방향을 1장으로 정리
  - **Accountability Chart**: 누가 뭘 책임지는지 명확화
  - **Scorecard**: 주간 5~15개 핵심 지표 추적
  - **Rocks**: 90일 최우선 과제 3~7개
  - **Level 10 Meeting**: 매주 90분, 정해진 아젠다로 운영
  - **IDS**: 문제를 Identify(식별) → Discuss(논의) → Solve(해결)
- **강점**: 전체 회사를 하나의 시스템으로 잡아줌. 구조가 명확하고 반복 가능

#### OKR (Objectives and Key Results)
- **창시자**: Andy Grove (Intel), Google이 대중화
- **구조**: Objective(야심찬 정성 목표) + Key Results(측정 가능한 정량 결과 2~5개)
- **주기**: 보통 분기 단위
- **강점**: 전략적 정렬, 유연성, 야심찬 목표 설정에 좋음
- **약점**: 실행 구조가 약함. "어떻게 할지"는 따로 정해야 함

#### Scaling Up (Rockefeller Habits)
- **창시자**: Verne Harnish
- **핵심**: 미팅 리듬(일간 허들 5~15분, 주간 60~90분, 분기 오프사이트)
- **도구**: One-Page Strategic Plan(OPSP), Critical Number, Quarterly Theme
- **강점**: 미팅 리듬이 매우 체계적
- **약점**: 10인 이상 팀에 최적화. 1인에겐 과함

#### E-Myth (The E-Myth Revisited)
- **창시자**: Michael Gerber
- **핵심 개념**: 사업자의 3가지 인격 — Entrepreneur(비전가), Manager(관리자), Technician(실무자)
- **프랜차이즈 프로토타입**: 사장 없이도 돌아가는 시스템을 만들어라
- **프로세스 문서화**: 모든 업무를 매뉴얼화 → 누구든(또는 AI든) 따라할 수 있게
- **강점**: AI 에이전트 시스템과 철학이 정확히 일치. "사람(사장)에 의존하지 않는 시스템"
- **약점**: 프레임워크라기보다 철학. 구체적 실행 도구가 부족

#### 4DX (4 Disciplines of Execution)
- **창시자**: FranklinCovey
- **4가지 원칙**:
  1. **WIG에 집중**: 가장 중요한 목표 1~2개만
  2. **Lead Measures 행동**: 결과(lag)가 아닌 행동(lead)을 추적
  3. **Scoreboard 운영**: 누구나 볼 수 있는 점수판
  4. **주간 책임 미팅**: 매주 서로 약속하고 점검
- **강점**: "일상 업무(소용돌이) 속에서 핵심 목표를 실행"하는 데 특화
- **약점**: 전체 회사 시스템이 아닌 "목표 실행"에만 집중

#### Holacracy
- **구조**: 직급 없음. Circle(서클) 단위로 자율 운영, 역할 기반
- **미팅**: Governance Meeting(구조 변경) + Tactical Meeting(운영)
- **강점**: 자율성, 빠른 역할 조정
- **약점**: 학습 곡선 높음, 5인 미만 팀에는 과도한 프로세스, 1인 CEO 구조와 맞지 않음

## 2. 추천: EOS + E-Myth 하이브리드 (리안 컴퍼니 맞춤)

### 왜 이 조합인가

| 이유 | 설명 |
|------|------|
| **EOS가 "뼈대"** | 목표(Rocks), 지표(Scorecard), 미팅(Level 10), 문제해결(IDS) — 회사 운영의 전체 구조를 잡아줌 |
| **E-Myth가 "철학"** | "사장이 없어도 돌아가는 시스템" = AI 에이전트 회사의 존재 이유 그 자체 |
| **1인 CEO에 최적** | EOS는 원래 10~250명 회사용이지만, 핵심 도구(V/TO, Scorecard, Rocks, IDS)는 1인도 쓸 수 있음 |
| **AI 에이전트와 궁합** | 프로세스 문서화(E-Myth) + 명확한 책임 배분(EOS Accountability Chart) = AI 에이전트가 가장 잘하는 것 |
| **세션 일관성 해결** | 매 세션마다 다른 AI가 시스템을 바꾸는 문제 → EOS의 "Core Processes 문서화"로 해결. 프로세스가 문서에 있으면 누가 와도 같은 방식으로 실행 |
| **4DX 요소 흡수** | WIG(가장 중요한 목표)와 Lead Measures(선행지표) 개념을 Rocks와 Scorecard에 녹여 넣음 |

### 왜 다른 것은 탈락했나

- **OKR 단독**: 실행 구조가 없음. "목표는 세웠는데 어떻게?"가 빠져있음
- **Scaling Up**: 좋지만 일간 허들, 월간 미팅 등 AI 에이전트에게 불필요한 계층이 많음
- **Holacracy**: 자율 조직은 좋지만, CEO가 1명인 구조에서 서클/거버넌스는 오버엔지니어링
- **4DX 단독**: 목표 실행에는 좋지만 회사 전체 운영 시스템이 아님

## 3. 리안 컴퍼니에 바로 적용하는 구체적 방법

### 3-1. Vision/Traction Organizer (V/TO) — 회사 방향 1장 정리

현재 `company_context.md`가 이 역할을 부분적으로 하고 있음. 여기에 EOS V/TO 구조를 입히면:

```
[리안 컴퍼니 V/TO]

Core Values (핵심 가치):
1. 실행 > 완벽 (빠르게 만들고, 빠르게 고친다)
2. 데이터 기반 판단 (감이 아니라 숫자)
3. 자동화 우선 (반복 작업은 시스템이)

Core Focus (핵심 집중):
- 마케팅 대행 (온/오프라인) + SaaS 개발

10-Year Target: 마케팅 AI 자동화 No.1
3-Year Picture: 월 매출 1,000만원, 고정 클라이언트 20개
1-Year Plan: 뷰티샵/카페 마케팅 대행 정착, 클라이언트 5개 확보

분기 Rocks (이번 90일):
  1. [Rock 1] 뷰티샵 레퍼런스 클라이언트 1개 확보 (Before/After)
  2. [Rock 2] 온라인 마케팅 파이프라인 자동화 완성
  3. [Rock 3] 월간 운영 루프 안정화 (콘텐츠 자동 생성)
```

### 3-2. Accountability Chart — 각 팀의 역할/목표 설정

현재 `회사 조직도.md`를 EOS Accountability Chart 형식으로 업그레이드:

| 팀 | 책임자(리드 에이전트) | 핵심 책임 3가지 | 분기 Rock |
|----|---------------------|----------------|-----------|
| **이사팀** | 준혁 (Opus) | GO/NO-GO 판단, 전략 검증, 리스크 관리 | 신규 프로젝트 2개 검토 완료 |
| **오프라인마케팅팀** | 검증자 (Opus) | 영업 자료 제작, DM/스크립트, 현장 검증 | 레퍼런스 클라이언트 1개 확보 |
| **온라인영업팀** | 한총괄 | 잠재고객 발굴, 제안서 작성, 클로징 | 영업 파이프라인 10건 확보 |
| **온라인납품팀** | 김태리 | SEO 콘텐츠, SNS 운영, 성과 분석 | 클라이언트 콘텐츠 납품 사이클 1회 완료 |
| **온라인마케팅팀** | 강하린 | 리드 생성, 광고 운영, 그로스 분석 | 리드 획득 단가 측정 체계 구축 |
| **뷰티샵전문팀** | (해당 리드) | 뷰티샵 특화 마케팅, 업종 지식 | 뷰티샵 3곳 컨택 |
| **카페전문팀** | (해당 리드) | 카페 특화 마케팅, 업종 지식 | 카페 3곳 컨택 |

**GWC 체크 (Gets it, Wants it, Capacity)**:
- 각 에이전트가 자기 역할을 "이해하는가(G), 원하는가(W), 할 수 있는가(C)" → 이건 프롬프트 설계 + 모델 선택으로 결정됨
- 예: 전략은 GPT-4o(Gets it), 검증은 Gemini(다른 관점), 실행은 Sonnet(빠름)

### 3-3. Scorecard — 주간 핵심 지표 추적

`보고사항들.md`를 Scorecard 형식으로 구조화:

```
[주간 Scorecard — 2026-W14]

| 지표 | 담당팀 | 목표 | 실적 | 상태 |
|------|--------|------|------|------|
| 영업 DM 발송 수 | 온라인영업팀 | 20건 | 15건 | 🟡 |
| 답장/관심 표현 | 온라인영업팀 | 5건 | 3건 | 🟡 |
| 블로그 발행 | 온라인납품팀 | 3건 | 3건 | 🟢 |
| 인스타 게시물 | 온라인납품팀 | 5건 | 5건 | 🟢 |
| 제안서 발송 | 온라인영업팀 | 3건 | 1건 | 🔴 |
| 클라이언트 미팅 | 오프라인마케팅 | 2건 | 0건 | 🔴 |
| 시스템 오류/장애 | 전체 | 0건 | 0건 | 🟢 |

Lead Measures (선행지표 — 내가 통제 가능한 행동):
- 이번 주 콘텐츠 생산량: X건
- 이번 주 아웃리치 수: Y건

Lag Measures (후행지표 — 결과):
- 이번 주 매출: 0원
- 이번 주 신규 클라이언트: 0개
```

### 3-4. Level 10 Meeting — 주간 리뷰 구조

리안이 매주 1회(15~30분) 확인하는 구조화된 리뷰:

```
[주간 Level 10 — 리안 컴퍼니 버전]

1. Good News (2분)
   → 이번 주 잘된 것 (자동 요약)

2. Scorecard 리뷰 (3분)
   → 🔴 항목 중심으로 확인

3. Rock 진척도 (3분)
   → 각 Rock On Track / Off Track 표시

4. Headlines (2분)
   → 클라이언트 반응, 시장 변화, 특이사항

5. To-Do 리뷰 (2분)
   → 지난주 약속한 것 완료 여부

6. IDS — Issues (15분)
   → 🔴 항목을 IDS로 처리:
   - Identify: 뭐가 문제?
   - Discuss: 원인은?
   - Solve: 다음 주 액션은?

7. 마무리 (1분)
   → 다음 주 To-Do 확정
```

**자동화 포인트**: 1~5번은 AI가 자동으로 `보고사항들.md`에서 생성 가능. 리안은 6번(IDS)에만 집중하면 됨.

### 3-5. 90일 Rock 사이클 — 분기 운영

```
[분기 사이클]

Week 0: 분기 계획
  - 지난 분기 Rock 달성률 리뷰
  - 새 분기 Rock 3~7개 설정
  - 각 팀에 Rock 배분
  - Scorecard 지표 조정

Week 1~12: 실행
  - 매주 Level 10 (위 구조)
  - Scorecard 자동 업데이트
  - 🔴 3주 연속 → 자동 경고 + IDS 강제 트리거

Week 13: 분기 리뷰
  - Rock 완료율 체크 (80% 이상이면 성공)
  - 다음 분기 Rock 준비
  - V/TO 업데이트 필요한지 확인
```

### 3-6. Core Processes 문서화 — 세션 일관성 해결

**현재 문제**: 매 세션마다 다른 AI가 와서 시스템을 제각각 바꿈

**EOS 해법**: Core Processes를 문서화하면 누가 와도 동일하게 실행

```
[리안 컴퍼니 Core Processes — 변경 불가 프로세스]

1. 영업 프로세스
   잠재고객 발굴 → 진단서 작성 → 아웃리치(DM/이메일) → 제안서 → 미팅 → 클로징
   * 각 단계의 템플릿은 knowledge/base/에 고정

2. 납품 프로세스
   클라이언트 온보딩 → 콘텐츠 전략 수립 → 콘텐츠 제작 → 클라이언트 확인 → 발행 → 성과 리포트
   * 주기: 주 1회 리포트

3. 콘텐츠 생산 프로세스
   트렌드 리서치 → 주제 선정 → 초안 작성 → 검수 → 발행
   * ops_loop.py가 자동 실행

4. 신규 프로젝트 프로세스
   아이디어 → main.py(이사팀) → GO/NO-GO → build_team.py → 팀 실행
   * 이 순서는 절대 변경 불가

5. 시스템 변경 프로세스
   문제 감지 → 변경 제안서 작성 → 리안 컨펌 → 실행 → 검증
   * AI가 독단적으로 시스템 구조를 바꾸지 못함
```

### 3-7. IDS 문제 해결 — 실전 예시

```
[Issue] 영업 DM 답장률이 5% 미만

Identify (식별):
- Scorecard에서 3주 연속 🔴
- 답장 5건 중 실제 관심 1건

Discuss (논의):
- DM 내용이 너무 일반적? → 온라인납품팀 검토
- 타겟이 잘못됐나? → 온라인영업팀 재분석
- 경쟁사는 어떻게 하나? → Perplexity 리서치

Solve (해결):
- Action: 이번 주까지 DM 템플릿 3종 A/B 테스트
```
