## 1. ChatDev 2.0 (DevAll)

### 개요
ChatDev 2.0은 YAML 기반의 "Zero-Code" 멀티에이전트 오케스트레이션 플랫폼으로,
노드(에이전트)와 엣지(연결)로 구성된 DAG(방향성 비순환 그래프) + 사이클 지원 그래프를 실행한다.

### 핵심 파일
- `workflow/graph.py` — 그래프 실행 엔진 (GraphExecutor)
- `workflow/runtime/execution_strategy.py` — DAG/Cycle/MajorityVote 전략 패턴
- `yaml_instance/ChatDev_v1.yaml` — 실제 소프트웨어 개발 워크플로우 정의
- `yaml_instance/reflexion_product.yaml` — 반성/피드백 루프 예시

### 핵심 패턴 3가지

#### 패턴 1: YAML 기반 조건부 엣지 라우팅
노드 간 전환은 모두 엣지(edge)에 `condition` 필드로 정의된다.
조건이 충족될 때만 다음 노드로 메시지가 전달된다.

```yaml
# ChatDev_v1.yaml에서 발췌
- from: Programmer Code Complete
  to: Code Review Comment Phase Prompt for Assistant
  trigger: true
  condition:
    type: keyword
    config:
      any:
        - <INFO> FINISHED   # 이 키워드가 출력에 있을 때만 다음 단계로
      case_sensitive: true
```

키워드 매칭, 정규표현식, 커스텀 함수 조건 모두 지원.

#### 패턴 2: loop_counter 노드로 피드백 루프 제어
무한 루프 방지와 반복 피드백을 동시에 구현한다.
`max_iterations` 소진 시 자동으로 다음 단계로 강제 진행한다.

```yaml
- id: Code Review Phase Loop Counter
  type: loop_counter
  config:
    max_iterations: 10    # 최대 10번 코드 리뷰 반복
    reset_on_emit: true
```

이 패턴으로 "Code Review → Programmer Fix → Code Review" 사이클을 구현.

#### 패턴 3: subgraph를 통한 재사용 가능한 워크플로우 모듈화
복잡한 로직을 별도 YAML 파일로 분리하고 `type: subgraph`로 호출한다.

```yaml
# reflexion_product.yaml에서 발췌
- id: Reflexion Loop
  type: subgraph
  config:
    type: file
    config:
      path: "subgraphs/reflexion_loop.yaml"  # 반성 루프를 독립 모듈로 관리
```

`subgraphs/reflexion_loop.yaml`, `subgraphs/react_agent.yaml` 등이 재사용 가능한 모듈로 존재한다.

### liancp에 바로 적용할 수 있는 것

1. **키워드 조건 라우팅을 이사팀 파이프라인에 적용**
   현재 `main.py`는 순차적으로 모든 에이전트를 실행하지만,
   준혁의 GO/NO-GO 판단에서 "NO-GO" 키워드 감지 시 즉시 중단하는 조건을 추가할 수 있다.

2. **loop_counter 패턴을 UltraProduct Wave에 적용**
   현재 Wave 4(QA 리뷰)에 반복 횟수 상한이 없다.
   `max_iterations: 3` 같은 제한을 두면 무한 루프를 방지할 수 있다.

3. **subgraph 패턴으로 팀 모듈화**
   각 팀(이사팀, 교육팀, 마케팅팀)을 독립 YAML/Python 모듈로 관리하면
   재조합이 쉬워진다.

### liancp보다 못한 점

- **멀티 LLM 미지원**: ChatDev는 단일 LLM 프로바이더(OpenAI)를 기본 상정한다.
  liancp처럼 Claude/GPT/Gemini/Perplexity를 역할별로 다르게 쓰는 구조가 없다.
- **비즈니스 의사결정 부재**: 코드/소프트웨어 개발에 특화되어 있고,
  시장조사 → 전략 → GO/NO-GO 같은 비즈니스 기획 파이프라인이 없다.
- **오프라인 세계 연결 없음**: liancp의 오프라인 영업팀, 분석팀 같은
  실세계 접점 에이전트 개념이 없다.

## 2. MetaGPT

### 개요
MetaGPT는 소프트웨어 회사의 SOP(표준 운영 절차)를 AI 에이전트로 구현한 프레임워크다.
"Code = SOP(Team)"이 핵심 철학이며, 메시지 기반 pub/sub 환경(Environment)에서 역할들이 상호작용한다.

### 핵심 파일
- `metagpt/roles/role.py` — 역할 기본 클래스 (observe → think → act 사이클)
- `metagpt/team.py` — Team 오케스트레이터
- `metagpt/environment/base_env.py` — 메시지 라우팅 환경
- `metagpt/roles/product_manager.py` — ProductManager 역할 (WritePRD 액션 포함)
- `metagpt/actions/write_prd.py` — PRD 자동 작성 액션

### 핵심 패턴 3가지

#### 패턴 1: Observe-Think-Act (OTA) 사이클
각 역할(Role)은 독립적으로 `_observe → _think → _act` 루프를 반복한다.
역할 간 결합이 없고, 메시지 큐를 통해서만 통신한다.

```python
# metagpt/roles/role.py에서 발췌
async def run(self, with_message=None) -> Message | None:
    if not await self._observe():   # 1) 내 관심사 메시지가 있는가?
        return                      # 없으면 대기
    rsp = await self.react()        # 2) think + act
    self.publish_message(rsp)       # 3) 결과를 환경에 발행
    return rsp
```

각 역할은 `_watch`로 자신이 구독하는 액션 타입을 선언한다.
예: ProductManager는 `UserRequirement` 메시지를 watch하고,
Architect는 `WritePRD` 결과를 watch한다.

#### 패턴 2: Action 단위로 역할 기능 분리 + `cause_by` 태깅
메시지에는 어떤 액션이 생성했는지 `cause_by` 태그가 붙는다.
이 태그가 메시지 라우팅의 핵심이다.

```python
# metagpt/actions/write_prd.py에서 발췌
return AIMessage(
    content="PRD is completed. ...",
    instruct_content=AIMessage.create_instruct_value(kvs=kvs, class_name="WritePRDOutput"),
    cause_by=self,  # WritePRD 액션이 이 메시지를 생성했음을 표시
)
```

ProductManager가 WritePRD를 완료하면,
Architect는 자신이 watch하는 `WritePRD` cause_by를 가진 메시지를 수신하고 반응한다.

#### 패턴 3: RoleReactMode — 역할별 실행 전략 선택
역할마다 다른 실행 전략을 선택할 수 있다:
- `REACT`: ReAct 논문 방식 (think → act → think → act 반복)
- `BY_ORDER`: 액션을 정해진 순서대로 실행
- `PLAN_AND_ACT`: 먼저 계획을 세우고, 계획에 따라 액션 실행

```python
# metagpt/roles/role.py에서 발췌
class RoleReactMode(str, Enum):
    REACT = "react"
    BY_ORDER = "by_order"
    PLAN_AND_ACT = "plan_and_act"
```

ProductManager는 `BY_ORDER`로 PrepareDocuments → WritePRD 순서를 보장한다.

### liancp에 바로 적용할 수 있는 것

1. **WritePRD 패턴을 이사팀 → UltraProduct 연결에 직접 활용 (현재 구멍 #1, #2 해결)**
   MetaGPT의 `write_prd.py`는 신규 요구사항/업데이트/버그픽스를 자동 분류하고
   JSON 구조화된 PRD를 생성한다. 이사팀 GO 판정 후 준혁의 출력을 입력으로 받아
   지훈(PRD 에이전트)이 동일한 패턴으로 PRD를 생성하면 된다.

   ```python
   # liancp에 적용할 수 있는 패턴
   # company/agents/jihun.py (새로 만들거나 복원)
   class PRDWriter:
       async def run(self, strategy_output: str) -> dict:
           # MetaGPT WritePRD와 동일한 구조
           prd = {
               "project_name": ...,
               "user_stories": ...,
               "competitive_analysis": ...,
               "tech_requirements": ...,
               "success_metrics": ...,
           }
           return prd
   ```

2. **cause_by 태깅 시스템으로 팀 간 핸드오프 추적**
   현재 liancp는 팀 간 결과물 전달이 파일 저장에 의존한다.
   MetaGPT처럼 메시지에 `cause_by` 태그를 붙이면 파이프라인 디버깅이 쉬워진다.

3. **`_watch` 패턴으로 에이전트 구독 선언**
   각 에이전트가 어떤 이전 단계 결과를 트리거로 실행되는지를
   명시적으로 선언하면 파이프라인 흐름이 자명해진다.

### liancp보다 못한 점

- **비즈니스 기획 파이프라인 없음**: 시장조사, 경쟁사 분석, GO/NO-GO 같은
  비즈니스 의사결정 레이어가 없다. 소프트웨어 요구사항 → 코드에 특화.
- **멀티 LLM 혼합 어려움**: 모든 역할이 동일한 LLM 설정을 공유하는 경향.
  liancp처럼 역할별로 다른 AI(GPT/Claude/Gemini)를 사용하는 구조가 복잡하다.
- **배포 후 피드백 루프 없음**: 제품 출시 후 사용자 피드백을 받아 다시 파이프라인에
  주입하는 메커니즘이 없다. (현재 구멍 #3)

## 3. crewAI

### 개요
crewAI는 Python 데코레이터 기반의 이벤트 주도(event-driven) 멀티에이전트 프레임워크다.
두 가지 레이어로 구성된다:
- **Crew**: 자율적 에이전트 협업 (역할 기반, 동적 위임)
- **Flow**: 정밀한 상태 관리와 조건부 라우팅 (프로덕션 환경)

### 핵심 파일
- `lib/crewai/src/crewai/flow/flow.py` — Flow 실행 엔진 (@start, @listen, @router 데코레이터)
- `lib/crewai/src/crewai/crew.py` — Crew 클래스 (에이전트 팀 실행)
- `lib/crewai/src/crewai/task.py` — Task 클래스 (guardrail, context 포함)
- `lib/crewai/src/crewai/memory/unified_memory.py` — 통합 메모리 (LLM 분석 + 벡터 검색)

### 핵심 패턴 3가지

#### 패턴 1: @start / @listen / @router 데코레이터로 이벤트 주도 파이프라인
Python 데코레이터만으로 복잡한 파이프라인 흐름을 선언적으로 정의한다.

```python
# crewai/flow/flow.py에서 발췌 + README 예시
class AdvancedAnalysisFlow(Flow[MarketState]):
    @start()
    def fetch_market_data(self):
        self.state.sentiment = "analyzing"
        return {"sector": "tech"}

    @listen(fetch_market_data)      # fetch_market_data 완료 후 자동 트리거
    def analyze_with_crew(self, market_data):
        crew = Crew(agents=[...], tasks=[...])
        return crew.kickoff(inputs=market_data)

    @router(analyze_with_crew)      # 결과에 따라 분기
    def determine_next_steps(self):
        if self.state.confidence > 0.8:
            return "high_confidence"
        return "low_confidence"

    @listen("high_confidence")      # 특정 라우터 결과에만 반응
    def execute_strategy(self):
        ...

    @listen(or_("medium_confidence", "low_confidence"))  # OR 조건
    def request_additional_analysis(self):
        ...
```

이 패턴은 liancp의 `main.py` 파이프라인을 완벽하게 대체할 수 있다.

#### 패턴 2: Pydantic 기반 공유 State로 팀 간 데이터 전달
모든 팀(Crew/Flow 메서드)이 동일한 `State` 객체를 공유한다.
파일 I/O 없이 메모리 내 타입 안전한 데이터 전달이 가능하다.

```python
class ProjectState(BaseModel):
    idea: str = ""
    market_analysis: dict = {}
    strategy: dict = {}
    go_decision: bool = False
    prd: dict = {}           # GO 후 기획 단계
    feedback_history: list = []  # 배포 후 피드백 루프

class LiancpFlow(Flow[ProjectState]):
    @start()
    def clarify_idea(self):
        self.state.idea = ...

    @listen(clarify_idea)
    def market_research(self):
        self.state.market_analysis = ...

    @router(market_research)
    def go_no_go(self):
        if go:
            return "approved"
        return "rejected"

    @listen("approved")
    def write_prd(self):       # 현재 구멍 #1 해결
        self.state.prd = ...
```

#### 패턴 3: Task Guardrail로 출력 품질 자동 검증
Task에 `guardrail` 함수를 붙이면 에이전트 출력이 기준 미달일 때 자동으로 재시도한다.

```python
# crewai/task.py 기능
def validate_prd_quality(output: TaskOutput) -> tuple[bool, str]:
    if len(output.raw) < 500:
        return (False, "PRD가 너무 짧습니다. 더 상세하게 작성하세요.")
    if "success_metric" not in output.raw:
        return (False, "성공 지표(KPI)가 없습니다.")
    return (True, "")

prd_task = Task(
    description="PRD 작성",
    agent=prd_writer,
    guardrail=validate_prd_quality,  # 자동 품질 검증
    max_retries=3
)
```

이 패턴은 현재 liancp에 없는 출력 품질 보장 메커니즘이다.

### liancp에 바로 적용할 수 있는 것

1. **Flow 패턴으로 main.py 전체 재구성 (현재 구멍 #1, #2 동시 해결)**
   현재 `main.py`의 순차 실행을 `@start/@listen/@router` 패턴으로 교체하면
   GO/NO-GO 분기, PRD 단계 삽입, UltraProduct 연결이 자연스럽게 구현된다.

2. **Pydantic State로 팀 간 핸드오프 표준화**
   현재 팀 간 데이터 전달이 파일 저장/로드에 의존한다.
   `ProjectState` 객체 하나로 전체 파이프라인 데이터를 관리하면
   이사팀 → PRD팀 → UltraProduct 연결이 타입 안전해진다.

3. **Unified Memory로 배포 후 피드백 루프 구현 (현재 구멍 #3 해결)**
   crewAI의 Memory는 LanceDB + LLM 분석으로 동작한다.
   사용자 피드백, 마케팅 결과, QA 결과를 Memory에 저장하면
   다음 프로젝트 시작 시 관련 기억을 자동으로 검색한다.

4. **Task Guardrail로 각 에이전트 출력 품질 보장**
   준혁의 GO/NO-GO 판단, 지훈의 PRD, CTO의 기술 스택 결정 등에
   guardrail을 붙이면 품질이 일정 수준 이하일 때 자동 재시도된다.

### liancp보다 못한 점

- **실시간 시장조사 에이전트 없음**: Perplexity 같은 실시간 검색 특화 에이전트
  개념이 없다. liancp의 서윤(Perplexity) 같은 역할은 커스텀 Tool로 구현해야 한다.
- **오프라인 세계 연결 없음**: liancp의 오프라인 영업팀, 이미지/영상 분석팀 같은
  실세계 접점 에이전트 개념이 없다.
- **Pain 발굴 단계 없음**: 아이디어 이전에 Pain을 발굴하는 단계(현재 구멍 #4)가 없다.
  liancp가 이 부분에서 선구적이다.

## 4. 종합 분석

### 3개 프레임워크 공통 베스트 프랙티스

#### BP-1: 조건부 라우팅 (모든 프레임워크에서 구현)
| 프레임워크 | 구현 방식 |
|-----------|----------|
| ChatDev | 엣지의 `condition.type: keyword` |
| MetaGPT | `cause_by` 태그 기반 메시지 필터링 |
| crewAI | `@router` 데코레이터 + 반환값 |

**공통 원칙**: 에이전트 출력 내용에 따라 파이프라인 흐름이 달라져야 한다.
단순 순차 실행이 아니라, 출력 결과에 반응하는 동적 라우팅이 필수다.

#### BP-2: 피드백 루프 / 반복 개선 (모든 프레임워크에서 구현)
| 프레임워크 | 구현 방식 |
|-----------|----------|
| ChatDev | `loop_counter` 노드 + 키워드 종료 조건 |
| MetaGPT | `max_react_loop` + `PLAN_AND_ACT` 모드 |
| crewAI | `max_retries` + `guardrail` 함수 |

**공통 원칙**: 에이전트는 한 번에 완벽한 결과를 내지 않는다.
반복 개선 루프와 최대 반복 횟수 제한이 반드시 필요하다.

#### BP-3: 구조화된 출력 + 타입 안전한 핸드오프 (모든 프레임워크에서 구현)
| 프레임워크 | 구현 방식 |
|-----------|----------|
| ChatDev | YAML 변수 + 엣지 `carry_data` |
| MetaGPT | `instruct_content` Pydantic 모델, `ActionNode` |
| crewAI | `output_pydantic`, `Flow[State]` 공유 상태 |

**공통 원칙**: 에이전트 간 데이터는 자유 텍스트가 아니라 구조화된 스키마로 전달되어야 한다.

#### BP-4: 역할 기반 에이전트 정체성 + 전문화
| 프레임워크 | 구현 방식 |
|-----------|----------|
| ChatDev | `role` 프롬프트 필드 (YAML) |
| MetaGPT | `name`, `profile`, `goal`, `constraints` 필드 |
| crewAI | `role`, `goal`, `backstory` 필드 |

**공통 원칙**: 에이전트는 자신의 역할, 목표, 제약, 페르소나를 명시적으로 부여받아야 한다.
단순히 시스템 프롬프트를 주는 것이 아니라, 구조화된 역할 정의가 필요하다.

import anthropic
import json

class PRDWriter:
    """지훈 — PRD 작성 에이전트 (MetaGPT WritePRD 패턴 차용)"""
    
    def __init__(self, api_key: str):
        self.client = anthropic.Anthropic(api_key=api_key)
    
    async def run(self, strategy_output: dict, go_reason: str) -> dict:
        prompt = f"""
당신은 제품 기획자입니다.
다음 전략 분석 결과를 바탕으로 PRD(제품 요구사항 문서)를 작성하세요.

## 전략 요약
{json.dumps(strategy_output, ensure_ascii=False, indent=2)}

## GO 판단 이유
{go_reason}

## 출력 형식 (JSON)
{{
  "project_name": "...",
  "problem_statement": "해결하려는 Pain은 무엇인가",
  "user_stories": ["As a ..., I want ..., so that ..."],
  "competitive_analysis": {{"강점": [], "약점": [], "기회": [], "위협": []}},
  "core_features": [{{"feature": "...", "priority": "P0/P1/P2", "description": "..."}}],
  "success_metrics": [{{"metric": "...", "target": "...", "timeframe": "..."}}],
  "tech_constraints": [],
  "timeline": {{"phase1": "...", "phase2": "...", "phase3": "..."}}
}}
"""
        response = self.client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=4096,
            messages=[{"role": "user", "content": prompt}]
        )
        content = response.content[0].text
        # JSON 파싱 시도
        try:
            return json.loads(content)
        except json.JSONDecodeError:
            # JSON 블록 추출 시도
            import re
            match = re.search(r'\{.*\}', content, re.DOTALL)
            if match:
                return json.loads(match.group())

### 권고 #3: 배포 후 피드백 루프 (현재 구멍 #3 해결)

**현재 상황**: 배포 후 피드백이 시스템에 재주입되지 않는다.

**권고 구현** (`company/knowledge/` 디렉토리 활용):

```python
# company/knowledge/feedback_loop.py (신규)
# crewAI의 Unified Memory 패턴 차용 (LanceDB 대신 JSON 파일 기반으로 단순화)

import json
from pathlib import Path
from datetime import datetime
from typing import Optional

class FeedbackLoop:
    """
    배포 후 피드백 → knowledge/base/에 저장 → 다음 프로젝트에서 자동 참조.
    crewAI의 Memory 패턴을 단순화하여 JSON 파일로 구현.
    """
    
    def __init__(self, base_dir: Path):
        self.feedback_dir = base_dir / "knowledge" / "feedback"
        self.feedback_dir.mkdir(parents=True, exist_ok=True)
        self.index_path = self.feedback_dir / "index.json"
        self._load_index()
    
    def _load_index(self):
        if self.index_path.exists():
            self.index = json.loads(self.index_path.read_text())
        else:
            self.index = []
    
    def record_deployment_feedback(
        self,
        project_name: str,
        feedback_type: str,  # "user_feedback", "marketing_result", "qa_issue"
        content: str,
        metrics: Optional[dict] = None
    ):
        """배포 후 피드백 기록 — liancp 비개발자 CEO가 직접 입력"""
        entry = {
            "id": f"{project_name}_{datetime.now().strftime('%Y%m%d_%H%M%S')}",
            "project": project_name,
            "type": feedback_type,
            "content": content,
            "metrics": metrics or {},
            "recorded_at": datetime.now().isoformat(),
        }
        
        # 프로젝트별 피드백 파일에 저장
        project_feedback_path = self.feedback_dir / f"{project_name}_feedback.json"
        if project_feedback_path.exists():
            feedbacks = json.loads(project_feedback_path.read_text())
        else:
            feedbacks = []
        feedbacks.append(entry)
        project_feedback_path.write_text(json.dumps(feedbacks, ensure_ascii=False, indent=2))
        
        # 인덱스 업데이트
        self.index.append({"id": entry["id"], "project": project_name, "type": feedback_type})
        self.index_path.write_text(json.dumps(self.index, ensure_ascii=False, indent=2))
        
        # 보고사항들.md에도 기록
        self._write_report(project_name, feedback_type, content)
    
    def get_relevant_feedback(self, new_project_description: str, top_k: int = 3) -> list:
        """
        새 프로젝트 시작 시 관련 과거 피드백 검색.
        (crewAI Memory의 semantic search를 단순 키워드 매칭으로 구현)
        """
        keywords = set(new_project_description.lower().split())
        scored = []
        for entry in self.index:
            project_feedback_path = self.feedback_dir / f"{entry['project']}_feedback.json"
            if project_feedback_path.exists():
                feedbacks = json.loads(project_feedback_path.read_text())
                for fb in feedbacks:
                    score = len(keywords & set(fb['content'].lower().split()))
                    if score > 0:
                        scored.append((score, fb))
        
        scored.sort(key=lambda x: x[0], reverse=True)
        return [fb for _, fb in scored[:top_k]]
    
    def _write_report(self, project_name: str, feedback_type: str, content: str):
        report_path = Path("보고사항들.md")
        timestamp = datetime.now().strftime("%Y-%m-%d %H:%M")
        report = f"\n\n## [{timestamp}] 피드백 기록 — {project_name}\n- **유형**: {feedback_type}\n- **내용**: {content[:200]}...\n"
        if report_path.exists():
            existing = report_path.read_text(encoding='utf-8')
            report_path.write_text(existing + report, encoding='utf-8')
```

## 7. 핵심 결론

3개 프레임워크 모두 liancp가 직관적으로 구현한 "다양한 AI 모델을 역할별로 분배"
패턴보다는 단일 LLM 중심이다. liancp의 멀티 AI 구조(Claude/GPT/Gemini/Perplexity)는
이 분야에서 선진적인 접근이다.

그러나 liancp가 부족한 부분은 명확하다:

1. **ChatDev에서 배울 것**: 조건부 라우팅과 루프 제어 (키워드 조건, loop_counter)
2. **MetaGPT에서 배울 것**: PRD 자동 생성 파이프라인과 구조화된 출력 스키마
3. **crewAI에서 배울 것**: Pydantic 기반 공유 State, Flow 패턴, 피드백 루프를 위한 Memory

가장 즉각적인 가치는 MetaGPT의 `WritePRD` 패턴을 `jihun.py`로 구현하고,
crewAI의 `Flow[State]` 패턴으로 `main.py`를 리팩토링하는 것이다.
이 두 가지만으로 현재 4개 구멍 중 3개(#1, #2, #3)를 해결할 수 있다.
