
### Best system prompt structures for embedding sales psychology (killer offer, price objection, scarcity) into LLM agents
### 최고 수준 시스템 프롬프트 구조: 판매 심리학 임베딩 (Killer Offer, Price Objection, Scarcity)

판매 심리학을 LLM 에이전트에 임베딩하는 **최적 프롬프트 구조**는 **적응형 few-shot + RAG 하이브리드**로, 고품질 예제 라이브러리를 동적으로 검색해 쿼리 유사성 기반 프롬프트 조합. 이 구조는 할루시네이션 40%↓, 변환율 2-3x↑ 사례 확인됨[3][6].

#### 1. **핵심 프레임워크: 4단계 적응형 구조 (RAG-Enhanced Few-Shot)**
```
[System Prompt Template]

1. **Killer Offer**: 가치 10x 과시 (구체 수치/증거). "이 오퍼로 [구체 혜택] 달성, 95% 고객 성공."
2. **Price Objection**: 가격=투자 프레임 전환. "한 달 커피값으로 [연 수익 5만 달러] 생성. ROI 20x."
3. **Scarcity**: FOMO 유발. "오늘 3자리 남음 / 24시간 한정 / [X]명 대기."

[Context Retrieval]: 쿼리 임베딩 → Vector DB (Qdrant)에서 top-3 유사 판매 예제/데이터 검색[1][6].
- 예: 쿼리 "비싸요" → Price objection 예제 불러옴.

User: [쿼리]

- 예시1: User: "너무 비싸." → Agent: "이 가격은 투자: 1개월 $99으로 연 $10K 절감. 지금 50% 할인, 2자리 남음[증거 링크]. 구매?"

```

### Negative rules in LLM prompts — how to write what NOT to do so agents reliably avoid bad outputs
### LLM 프롬프트 Negative Rules 핵심 프레임워크
**Negative rules(부정 규칙)**은 "하지 말라" 지시로 LLM의 환각, 편향, 일관성 오류를 80-95% 억제. 실전 성공률: OpenAI 테스트에서 명확 negative rules 적용 시 오류율 40%↓[3][4].

#### 1. 기본 구조: "Do NOT + 구체적 행동 + 이유"
- **틀린 예**: "잘못된 정보 말하지 마."
- **최적 예**: "Do NOT 생성된 사실을 발명하거나 추측. 모르면 '정보 부족'이라고만 응답."[3][4]
  - **효과**: 환각(hallucination) 70% 차단. LLM이 "자신 있게 틀린 답" 생성 방지[4].

#### 2. Top 5 Negative Rules 템플릿 (즉시 복붙 가능)
| 규칙 유형 | 템플릿 예시 | 억제 대상 | 성공 사례 (오류 ↓율) |
|----------|-------------|----------|---------------------|
| **환각 방지** | "Do NOT 알려지지 않은 사실 발명. 출처 없으면 '확인 불가' 출력." | 거짓 생성 | 85% (수학/사실 오류)[4] |
| **편향 차단** | "Do NOT 위치/길이/인종 편향 반영. 첫 번째/긴 답 선호 금지." | Position/Verbosity bias | 60% (AI 판정 오류)[2] |
| **일관성 확보** | "Do NOT 이전 응답과 모순. temperature=0 고정처럼 일관 출력." | 비결정성 | 75% (반복 질문 변동)[2] |
| **출력 형식 제한** | "Do NOT 추가 설명/서론/결론 추가. 단일 단어/JSON만 출력." | 과잉 생성 | 90% (라벨링 태스크)[3] |
| **안전 가드** | "Do NOT 성적/차별/폭력 콘텐츠 생성. 거부 시 '안전 정책 위반'." | 부적절 출력 | 95% (고객 LLM)[4] |

#### 3. 실전 적용 스텝 (3단계, 5분 내 구축)
1. **Identity 먼저**: "You are [역할]."[3]
2. **Positive rules**: "항상 [해야 할 일] 하라."
3. **Negative rules 삽입**: "Do NOT [금지 목록]. 이유: [오류 예시]."
   - **전체 예시 프롬프트** (제품 리뷰 분류, 오류 2% 미만)[3]:
     ```
     You are a sentiment analyzer.
     Instructions: Classify as Positive/Negative/Neutral.
     Do NOT 추가 텍스트/이유 설명. 단일 단어만 출력.
     Do NOT 환각: 리뷰 외 정보 사용 금지.
     Example: "Love it!" → Positive
     Review: [입력]
     ```

#### 4. 고급 팁: 에이전트 최적화 (성공률 95%+)
- **NSGA-II 영감 counterfactual**: 실패 시뮬 → "Do NOT [실패 행동]. 대신 [대안]."[1]
- **계층 우선**: Instructions 파라미터 > Input 프롬프트[3]. GPT-5: 정밀 negative 선호, Reasoning 모델: 고수준만[3].
- **테스트 지표**: 동일 질문 10회 → 일관성 95% 목표. 편향: A/B 비교[2].
- **한계 극복**: SFT 지식 불일치 시 "Do NOT 인간 모방 위해 발명"[2].

이 프레임워크로 에이전트 배포 시 bad output 90%↓. A/B 테스트: negative rules 유무 차이 3배[2][3][4].

### Few-shot example selection strategy for business agents: how many examples, what format, which edge cases to include
### **핵심 숫자: 2-5개 예시 최적**
비즈니스 에이전트(고객 지원, 세일즈 팔로업 등)에서 **few-shot 예시 수는 2-5개**가 표준. 1개(1-shot)로 기본 패턴 학습 가능하나 복잡 태스크는 3-5개로 성능 20-80% 향상[1][3][6]. 10개 초과 시 토큰 비용↑, 랜덤 선택 수준으로 회귀[4]. **A/B 테스트로 2 vs 4개 비교: 4개가 format fidelity 15%↑**[2].

### **포맷: Q&A 쌍 + 명확 구조**
**JSON-like 또는 명시적 역할(role) 지정**. 입력-출력 쌍으로 패턴 보여줌. 비즈니스 적용:
```
examples = [
  {"role": "user", "content": "고객: 장치 안 켜짐."},
  {"role": "assistant", "content": "10초간 전원 버튼 누르세요. 재시작 확인."}
]
```
- **비즈니스 템플릿 예: 세일즈 팔로업**[3]
  ```
  Input: "고객 우려: AI 로보틱, 경쟁사 3배 빠름. 다음: 데모."
  Output: "안녕 Sarah, 콘텐츠 병목 이야기 좋았어요. 경쟁사 3배 빠른데 AI 로보틱 걱정 이해해요. few-shot으로 당신 2-3개 샘플 주면 톤 맞춤. 화Tue 15분 데모 어때요?"
  ```
- **동적 삽입: 임베딩 유사도 top-3 선택**[2]. 요청 임베딩 → 벡터 인덱스 쿼리 → Q-A 포맷 유지. 로그로 A/B 측정.

**순서 팁: 가장 중요한 예시를 마지막에** – LLM이 최근 정보에 30-50% 가중[5].

### **포함 엣지 케이스: 30% 비중 할당**
**예시 중 1-2개(20-30%)를 엣지 케이스**로: 현실 대화 반영, 실패 패턴 방지. 비즈니스 에이전트 필수:
- **일반(70%)**: 표준 시나리오 (e.g., "기본 트러블슈팅").
- **엣지(30%)**:
  | 엣지 유형 | 예시 (고객 지원) | 예시 (세일즈) | 효과 |
  |-----------|------------------|---------------|------|
  | **부정/반대** | "여전히 안 됨." → "하드 리셋: 배터리 30분 충전 후." | "AI 로보틱 우려" → "few-shot으로 당신 톤 80% 매치." | objection handling 25%↑[3] |
  | **복잡 입력** | 긴 트랜스크립트 → 2-3 포인트 참조. | 기술 고객: "제네릭 안 됨" → "당신 5개 샘플로 테스트." | accuracy 40%↑[2] |
  | **톤 변형** | 캐주얼 vs 포멀. | 비세일지: "Game?" | 브랜드 보이스 90% 일치[1] |

**선택 전략: ACSESS (자동 조합)** – 20개 전략 중 top performer 자동 믹스. 저샷(2-3)에서 효과 극대, 데이터셋 따라 10-20% 성능↑[4].

### **실전 프레임워크: 4단계 배포**
1. **2-3개 정적 예시 시작**: format/tone 고정.
2. **동적 retrieval 도입**: 입력 drift 시 top-3 유사 예시[2].
3. **A/B 테스트**: 예시 세트 교체, 실트래픽 metric (response time, conversion)[2].
4. **측정 & 스케일**: 예시당 80% 퀄리티, 20% 시간 단축 목표[3]. 비용: fine-tuning 대비 70%↓[5].

### How to write business philosophy into AI agent instructions without losing specificity or becoming too abstract
### 핵심 프레임워크: **Agentic Compact + Autonomy Levels + Critical Anchoring**

**비즈니스 철학을 AI 에이전트 지시어에 삽입할 때, 명확한 목표(objectives), 테스트 가능한 제약(constraints), 동적 평가 루프를 구조화하라. 추상화 피하기 위해 구체적 API/임계값/승인 워크플로우로 분해하고, 철학을 '왜'가 아닌 '어떻게 실행'으로 변환.[1][2][3]**

#### 1. **철학 → 실행 분리 (Modular Design)**
   - **비즈니스 철학 예: "고객 만족 최우선"** → 추상 탈피: "고객 불만 시 discount API 호출, **최대 20% 할인 한도**, 매니저 승인 초과 시 pause & human handover."[1]
     - 사례: e-commerce 에이전트가 50% 할인 남발 방지 → revenue 보호 (실제 drift 사례: 만족도 ↑ but 수익 ↓).[1]
   - LLM은 언어/추론만, 비즈니스 로직(계산, DB 쿼리)은 별도 시스템으로 → 검증 가능성 ↑.[1]
   - **수치 지표**: 할인 임계값 20%, 승인 지연 <5분, drift 감지 시 95% 재평가 정확도 목표.[1]

#### 2. **Autonomy Levels로 specificity 유지 (Knight Framework 적용)[2]**
   | Autonomy Level | 철학 삽입 방법 | Specificity 유지 팁 | 사례 (고위험 워크플로우) |
   |----------------|----------------|---------------------|--------------------------|
   | **L1: Tool-Only** | 철학 무시, 사용자 주도. 에이전트는 제안만. | 모든 액션 user approve 필수. | 법률 검토: "합법성 확인 후 pause" → accountability 100%.[2] |
   | **L2: Planner** | 철학을 plan 초안에 embedding. User edit 후 실행. | Plan 단계별: "목표: 만족도 ↑, 제약: 비용 <10%". | 고객 지원: plan에 "할인 >15% 시 reject" hardcode.[2] |
   | **L3+: High Autonomy** | 철학을 intrinsic value로 (e.g., "utility max + moral guardrail"). | Memory 모듈에 drift detector: 3개월 후 행동 변화 >10% 시 재훈련.[1][2] |

   - **트레이드오프 수치**: Autonomy ↑ → 효율 +20-50%, but accountability -30% (고위험 시 L1/L2 추천).[2]

#### 3. **Critical Instruction Anchoring (ServiceNow 방법)[3]**
   - **비즈니스 규칙 고정**: "항상 규칙 1 우선 > 규칙 2" 순서 명시. Deviation 0% 보장.
     - 예: "철학: 비용 효율" → "**API 호출 전 예산 체크, 초과 시 no-action**".[3]
   - **지시어 템플릿**:
     ```
     [철학 요약]: 고객 중심 + 수익 보호.
     [Objectives]: 1. 불만 해결 (타겟: CSAT +15%). 2. 할인 총액 <월 예산 5%.
     [Tools & Bounds]: discount_api(max=20%), approval_workflow(threshold=3 tickets).
     [Drift Check]: 매 100 interaction 후 log review, deviation >5% 시 alert.
     [Fallback]: Unknown → human loop.
     ```

#### 4. **실전 배포 체크리스트 (Continuous Governance)[1]**
   - **초기 3 Checks**: Model provenance (출처 확인), Data grounding (회사 데이터 grounding), Objectives (명확 + testable).[1]
   - **동적 모니터링**: Memory 변화 추적 → preference drift 방지 (사례: 반복 작업 후 정치적 bias d=0.6).[4]
   - **평가 루프**: 실시간 adaptation + SME feedback. Reliability: 반복 태스크 일관성 70-75% 목표 (현재 업계 30-75%).[4]
   - **도구 추천**: Palmyra X5 스타일 – tool-calling <1초, boundary 내 99% 예측 가능.[1]

### Prompt compression techniques — how to keep system prompts under token limits while preserving strategic intent
**Prompt compression shrinks LLM inputs by 4-20x while retaining 95%+ strategic intent**, using hard/soft methods to fit token limits (e.g., 128K for GPT-4o).[1][2][4]

Apply these top techniques for real-world use—prioritize **hard methods** for compatibility without model changes.

### Core Frameworks: Hard vs. Soft
| Method | How It Works | Compression Ratio | Best For | Example Tools |
|--------|--------------|-------------------|----------|---------------|
| **Hard** (text-based, human-readable) | Filter/replace tokens; 2-stage: sentence removal + token compression. | 10-20x | Closed LLMs, inference speed. | LLMLingua (20x on GSM8K/BBH datasets, latency -50%).[4] |
| **Soft** (vector embeddings) | Encoder → short vectors → decoder; LoRA-tuned. | 4-500x | Open models, high ratios. | ICAE (512→32 tokens, 16x); 500xCompressor.[3] |

**Hard dominates production** (versatile, no LLM mods).[1][2]

### 실전 Top-5 Techniques (Token Budget <4K Target)
1. **LLMLingua (Budget-Aware Filtering)**: Small model (GPT2/LLaMA-7B) scores token relevance → remove 80% noise.  
   - **Apply**: `coarse → fine compression`; align via instruction tuning.  
   - **Metrics**: 20x ratio, 93% perf on ICL/reasoning (GSM8K: 85→82% acc).[4]  
   - Python: Use MS framework → compress ShareGPT convos 15x.

2. **LongLLMLingua (Subseq Recovery)**: Reorder docs + recover key spans → longer windows.  
   - **Gain**: 2x window vs. base; AdaComp adapts dynamically.[3]

3. **CPC/TCRA-LLM (Embedding Rank)**: Context embeddings rank sentences → summarize semantics.  
   - **Ratio**: 10x; preserves intent via vector sim (cosine>0.9).[3]

4. **Knowledge Distillation**: Train tiny model on large LLM outputs → mimic behavior.  
   - **Case**: Compress 5K→500 tokens; PEFT cuts params 99%.[2][5]

5. **Hybrid (Hard+Soft)**: Filter first → encode rest → 30x total.  
   - **Pitfall**: Sequential time; fix with parallel encoders.[3]

### Step-by-Step Framework (Compress Any Prompt)
```
1. Budget Set: Target = 20% original tokens.
2. Filter: Remove redundants (sentences <0.1 relevance score).[1]
3. Encode: Embed → cluster → replace (e.g., ICAE 16x).[3]
4. Validate: Test on 3 tasks; retain >95% output match (ROUGE/BLEU).
5. Iterate: Token-level refine for coherence.
```
**Benchmark**: LLMLingua beats GPT-4 summarizer by 2x ratio at same perf.[4]

### 수치 사례 (2024-2025 SOTA)
- **GSM8K**: 20x → 92% acc preserved.[4]
- **Arxiv Summ**: 15x → latency -60%.[4]
- **High-Ratio**: 500xCompressor → 5120 tokens fit 128 limit.[3]

**Limits**: Loses nuance at >50x; task-specific tuning needed (e.g., code prompts).[2][3] Scale by chaining (compress → LLM → recompress).[1]
