# CPO 분석 — 지역 소상공인 010번호 + 인스타 자동 수집 시스템

> Wave 1 | 작성: CPO | 날짜: 2026-03-24

---

## 제품 전략

리안 컴퍼니의 문자/DM 영업을 위한 리드 수집 단계를 사람 손 없이 자동화한다.
외부 SaaS 판매용이 아닌 내부 영업 인프라로, 빠른 동작 안정성과 엑셀 출력 품질이 전부다.

---

## 타겟 + Pain

- **타겟**: 리안 (리안 컴퍼니 CEO, 비개발자) — 매일 직접 소상공인 연락처를 수동 검색하거나 외주에 맡기는 상황
- **Pain**: 지역 + 업종 조합으로 소상공인 연락처를 모으는 작업이 반복적·수작업이다. 수집한 번호가 010인지 유선인지 구분도 직접 해야 하고, 폐업 업체 걸러내는 것도 수동이다. 결국 영업 시간의 상당 부분이 리스트 만들기에 사라진다.
- **아하 모먼트**: "의정부 카페"를 입력하고 실행 버튼을 눌렀더니 5분 뒤 010번호 + 인스타 계정이 정리된 엑셀이 바탕화면에 생겼을 때. 검증된 건과 폐업의심 건이 시트로 나뉘어 있어서 바로 문자 발송 리스트로 쓸 수 있을 때.

---

## MVP 범위 (Must Have)

1. **지역 + 업종 입력 → 수집 실행**
   카카오맵 API 기반 수집을 핵심 축으로 삼는다. 구글맵은 차단 가능성이 높으므로 2순위. 당근마켓·인스타·네이버블로그는 수집량 보조 역할로 제한한다.

2. **010 번호 필터링**
   유선(031/032/02 등), 050 안심번호, 1588/080 등 비모바일 번호를 자동 제거한다. 이 필터가 없으면 엑셀이 쓰레기가 된다. 핵심 중의 핵심.

3. **네이버플레이스 실존 검증**
   "확인됨 / 미확인 / 폐업의심" 3단계. 폐업의심 업체에 문자 보내면 리안 컴퍼니 이미지에 손상이 간다. 검증은 선택이 아니라 필수다.

4. **중복 제거**
   전화번호 기준 1차, 업체명 기준 2차. 5개 플랫폼에서 같은 업체가 중복 수집되는 것은 당연한 현상이므로 병합 로직이 반드시 있어야 한다.

5. **엑셀 내보내기 (시트 분리)**
   시트 구성: 확인됨 / 미확인 / 폐업의심 / 전체. 리안이 엑셀을 열면 바로 문자 발송 리스트로 쓸 수 있는 상태여야 한다.

6. **수집 진행 현황 실시간 표시**
   플랫폼별 진행 바 + 수집 건수 카운터. 작업이 멈춘 건지 돌아가는 건지 리안이 볼 수 있어야 한다. 없으면 불안해서 프로그램을 강제 종료한다.

7. **수집 이력 관리**
   날짜·지역·업종·수집량 기록. 어제 의정부 카페 했는지 안 했는지 기억하려면 이력이 필요하다. 1회 최대 500건 / 1일 최대 1000건 제한도 여기서 관리.

## MVP 제외 (나중에)

- **인스타그램 스크래핑 자동화**: 로그인 세션 관리, 차단 회피 로직이 복잡하다. MVP에서는 인스타 계정을 "있으면 좋은 것"으로 취급하고, 카카오맵/네이버 등에서 인스타 링크가 같이 나오면 수집하는 수준으로 제한한다. 인스타 전용 검색은 2차에서.

- **당근마켓 스크래퍼**: 당근은 구조가 자주 바뀌고 차단이 빠르다. 포천/의정부 성공 사례가 네이버 기반이었음을 참고. MVP에서는 보조 플랫폼으로만 시도, 실패해도 전체 수집에 영향 없도록 설계.

- **이력 비교 기능 (신규/삭제/변경 분석)**: 영업 리스트 고도화에 유용하지만 지금 당장 필요한 기능이 아니다. 2차.

- **PyInstaller 단독 실행 파일 패키징**: 개발 단계에서는 Python 직접 실행으로 충분하다. 안정성 확인 후 패키징.

- **업종 자동 분류 / 카테고리 태깅**: 수집된 업체를 AI로 분류하는 기능. 지금은 수집 자체가 먼저다.

## 성공 기준 (KPI)

- **30일 목표**: 포천/양주/의정부 3개 지역, 각 5개 이상 업종 조합으로 수집 실행 → 1회 실행당 순수 010번호 200건 이상 + 검증 완료율 80% 이상
- **핵심 지표**:
  - 엑셀 품질: 출력된 010번호 중 실제 운영 중인 업체 비율 (폐업의심 제외 후 수동 샘플링 10건으로 측정)
  - 수집 속도: 500건 수집 + 검증 완료까지 10분 이내
  - 오류율: 수집 실행 5회 중 치명적 오류(앱 크래시 / 엑셀 미생성) 0건

## 배포 블로커

- **카카오맵 API 차단**: 카카오맵은 공식 API를 쓰거나 Playwright로 우회해야 한다. 공식 API는 일 10만 건 무료로 충분하지만 키 발급 필요. CTO가 API vs 스크래핑 중 뭘 쓸지 결정해야 한다.
  - 해결 방법: 카카오 Local API 키 발급 (Kakao Developers 무료). 스크래핑 대신 공식 API 우선.

- **네이버플레이스 검증 속도**: 500건을 순차 검증하면 수십 분 걸린다. 비동기 처리 안 하면 사용자가 포기한다.
  - 해결 방법: asyncio로 동시 요청, 단 네이버 차단 방지를 위해 동시 요청 수 5 이하 제한 + 랜덤 딜레이.

- **인스타 수집 차단**: 인스타그램은 로그인 없이 접근하면 빠르게 429 또는 리다이렉트가 뜬다.
  - 해결 방법: MVP에서는 다른 플랫폼 수집 결과에 인스타 링크가 포함된 경우에만 가져오는 방식으로 범위 제한. 인스타 전용 검색은 제외.

- **Windows 환경 Playwright 설정**: Playwright chromium 설치 + Windows 경로 이슈가 생길 수 있다.
  - 해결 방법: requirements.txt에 playwright 포함, 최초 실행 시 `playwright install chromium` 자동 실행 스크립트 추가.

## CTO에게 요청

1. **카카오맵 수집 방법 확정**: 공식 Kakao Local API (추천) vs Playwright 스크래핑 중 선택해라. 공식 API는 안정적이고 차단 없지만 업종 카테고리 코드 매핑이 필요하다. 스크래핑은 유연하지만 차단 리스크 있다. 포천/의정부 성공 사례가 네이버 API 캡처 방식이었음을 참고해서 판단해라.

2. **네이버플레이스 검증 방식**: `map.naver.com/p/api/search/allSearch` 같은 내부 API 캡처 방식이 가능한지, 아니면 Playwright로 DOM 파싱해야 하는지. 기존 소상공인_실연락처_수집_자동화_시스템에서 이미 이 API로 포천 92건 수집 성공 사례가 있으니 그 방식 재활용 가능한지 확인해라.

3. **1일 1000건 제한 구현 위치**: 앱 레벨에서 SQLite 이력 조회로 막을 건지, 아니면 수집 워커 레벨에서 카운터로 막을 건지. 두 레이어 모두 적용 권장.

4. **PyQt6 GUI 최소 화면 크기 기준**: 리안이 Windows 데스크톱에서 쓰는 모니터 해상도 기준으로 진행 화면과 결과 테이블이 스크롤 없이 한 화면에 들어오는지 확인 필요.
