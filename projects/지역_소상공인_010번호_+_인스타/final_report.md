# 최종 리포트 — [지역 소상공인 010번호 + 인스타 수집 툴]

## 전체 평가: 조건부 준비 (CONDITIONAL READY)
**이유**: 핵심 수집 및 검증 로직은 훌륭하게 설계 및 테스트되었으나, **"비개발자 CEO(리안)가 직접 사용한다"는 핵심 전제를 배포 단계에서 망각**했습니다. 현재 상태로는 리안이 명령프롬프트(CMD)를 열고 파이썬 패키지를 설치해야 작동합니다. 또한, 대량 수집 시 발생할 수 있는 SQLite 동시성 문제와 엑셀 저장 실패에 대한 안전망이 부족합니다.

## Wave별 산출물 요약

| Wave | 담당 | 핵심 내용 및 평가 |
|---|---|---|
| **Wave 1** | CDO, CPO, CTO | **우수함.** 기획 의도(010 필터링, 폐업 필터링)가 명확하며, CTO의 Playwright 봇 차단 리스크 분석과 GUI 블로킹 방지 설계가 매우 현실적임. |
| **Wave 2** | CPO-CTO 합의, PM | **현실적 타협 성공.** 구글맵 기본 OFF, 네이버 실존 검증 선택 옵션 전환 등 속도와 안정성을 위한 타협이 적절히 이루어짐. |
| **Wave 4** | QA | **표면적 통과.** 2개의 자잘한 버그를 찾아 수정했으나, 실제 하드코어한 엣지케이스(네트워크 단절, 파일 잠금 등)에 대한 테스트가 누락됨. |
| **원본** | 아키텍처 | **경고.** CPO가 제외한 '이력 비교 기능'이 DB 모델(`HistoryService.compare`)에 남아있는 등 일부 스펙 크립(Scope Creep) 발생. |

## 발견된 엣지케이스 + 보완 필요

| 분류 | 엣지케이스 상황 | 현재 예상되는 결과 | 보완 요구사항 |
|---|---|---|---|
| **CRITICAL** | **엑셀 파일 열림 상태에서 저장 시도**<br>(사용자가 이전 결과를 엑셀로 열어둔 상태에서 다시 수집 완료) | `PermissionError` 발생. 10분간 수집한 **데이터가 허공으로 날아감.** | 예외 처리 후 임시 파일명(예: `result_1.xlsx`)으로 자동 저장 우회 로직 필수. |
| **CRITICAL** | **수집 중 네트워크 단절 또는 앱 강제 종료** | Playwright의 Headless 브라우저 프로세스가 메모리에 **좀비로 남음**. 누적 시 PC 멈춤. | 메인 윈도우 종료 시 `playwright.stop()` 및 강제 프로세스 킬(Kill) 로직 추가 필수. |
| **HIGH** | **중복 업체의 전화번호 충돌**<br>(카카오는 유선번호, 네이버는 010번호인 동일 업체) | `Deduplicator`의 병합 우선순위가 명확하지 않아 010 번호가 유선번호로 덮어씌워질 수 있음. | 병합 시 **무조건 010(모바일) 번호를 최우선으로 보존**하도록 규칙 하드코딩. |
| **HIGH** | **지역명 모호성**<br>(예: "광주" 입력 시 경기도 광주인지 전라도 광주인지) | Playwright가 엉뚱한 지역을 스크래핑하거나 타임아웃 발생. | API 요청 전 '시/도'가 포함되지 않은 짧은 지역명은 경고 팝업 띄우기. |
| **MEDIUM** | **일일 1000건 제한 우회** | 앱을 껐다 켜면 메모리 카운터가 초기화되어 계속 수집 가능. | SQLite DB의 `today()` 쿼리를 통해 일일 수집량을 체크하도록 강제. |

## 보안 및 안정성 최종 확인

| 항목 | 상태 | 코멘트 |
|---|---|---|
| **SQLite 동시성 (Locking)** | ⚠️ 위험 | QA 리포트에서 "WAL 모드 권장 (미적용)"이라고 명시함. QThread 여러 개가 동시에 DB에 Write 시도 시 `database is locked` 에러 무조건 터짐. **출시 전 WAL 모드 적용 필수.** |
| **API 키 노출** | ✅ 안전 | `.env`로 분리됨. 단, 비개발자인 리안이 `.env` 파일을 직접 만들고 세팅할 수 있는지 의문. |
| **캡차(Captcha) 대처** | ⚠️ 위험 | 카카오/네이버가 봇을 감지해 캡차를 띄우면 무한 대기(Hang)에 빠질 수 있음. 명시적인 타임아웃(Timeout) 설정 및 건너뛰기 로직 확인 필요. |

## 독립 검증자 종합 의견

**"설계는 날카롭고 개발은 깔끔하나, '누가 쓰는가'를 마지막에 잊었습니다."**

**좋은 점:** 
CTO와 CPO가 "무엇을 버려야 할지" 정확히 알고 타협한 점이 훌륭합니다. 특히 010 번호 필터링을 최우선으로 두고, 구글맵과 인스타 직접 수집의 리스크를 피해 우회로(간접 검색)를 택한 것은 매우 실무적인 결정입니다. 

**나쁜 점:** 
QA 요원이 너무 관대합니다. 코드 구조와 타입 힌트 같은 '개발자 관점'의 버그만 잡았고, 사용자가 엑셀 파일을 열어두거나 중간에 강제 종료하는 등의 '실사용자 관점'의 치명적 오류 테스트를 전혀 하지 않았습니다.

**가장 우선 해결할 것 3가지:**
1. **원클릭 실행기(`start.bat`) 제작**: 비개발자 CEO를 위해 Playwright 설치부터 앱 실행까지 한 번에 처리하는 스크립트가 절대적으로 필요합니다.
2. **엑셀 저장 실패 방어 로직**: 파일 잠금 시 데이터를 날리지 않도록 임시 파일 저장 백업을 추가해야 합니다.
3. **SQLite WAL 모드 즉시 적용**: DB 락(Lock) 에러를 막기 위해 데이터베이스 초기화 코드에 `PRAGMA journal_mode=WAL;`을 추가해야 합니다.
